# 当前迭代计划

> 更新日期：2026-07-06  
> 适用范围：比赛 MVP 开发期，主验证领域 `ai_app_dev`（人工智能应用开发实训）

## 1. 为什么调整原计划

原设计文档中的 9 周计划以“项目初始化 -> 数据模型与知识库 -> 诊断 -> Agent -> 可视化反馈 -> 报告评测”为主线，适合作为完整工程推进顺序。

但当前项目已经提前完成了部分前端演示闭环：

- Vue 前端应用壳、演示工作台、诊断测评、Agent 协作、学习资源、学习报告、指标页已形成可点击流程。
- 知识库管理页已支持单条手动导入知识点。
- 后端已有 `/api/v1` 路由骨架、统一响应、诊断、生成任务、资源、报告、知识库列表/检索/新增接口。
- 生成链路当前仍以 `demo_flow_service` 和规则化 MVP 图为主，LLM 双模型审核、真实异步任务和完整评测仍需补强。

因此迭代策略应调整为：

```text
前端演示闭环稳定 -> 后端真实能力补齐 -> 评测脚本量化 -> 演示材料打磨
```

目标不是重做计划，而是让后续开发围绕已经可见的产品流程继续填实。

## 3. 调整后的里程碑

### M0：前端演示闭环稳定（当前优先）

目标：保证评委和团队成员能从 `localhost:5173` 走完完整演示路线。

验收：

- 首页能清晰引导：诊断 -> Agent 协作 -> 资源 -> 反馈 -> 报告 -> 指标。
- 核心页面中文正常、无旧乱码、无营销落地页。
- 知识库页能新增知识点，并提示“待重建索引”。
- `npm run build` 通过。
- Docker 前端容器与本地访问地址一致，优先使用 `http://localhost:5173/`。

### M1：知识库与诊断数据补齐

目标：满足 MVP 数据规模和可解释性。

任务：

- 确认 `ai_app_dev` 至少 50 条真实知识点。
- 确认至少 60 道诊断题，选择题与简答题覆盖不同难度。
- 为知识点补齐分类、难度、标签、来源、内容。
- 实现知识变更后的索引重建流程：新增/修改知识点 -> 标记 `needs_reembedding` -> 重建 Chroma -> 清除标记。
- 知识关系表至少覆盖 prerequisite、dependent、related 三类关系。

验收：

- 领域配置页校验通过。
- 知识库检索能返回新增知识点。
- 手动导入后学习路径 `needs_refresh=true`。

### M2：后端闭环真实化

目标：把前端已经展示的流程逐步接成真实后端逻辑。

任务：

- 将 `demo_flow_service` 中的关键逻辑拆入正式 service。
- 生成任务支持真实异步状态流，SSE 事件来自任务状态和 `agent_runs`。
- Profile Analysis Agent、Knowledge Retrieval Agent、Content Generation Agent、Review and Validation Agent、Orchestrator Agent 输出结构化结果。
- 每个 Agent 写入 `agent_runs` 和 `agent_messages`。
- 资源生成必须包含来源引用。

验收：

- Agent 协作页展示的事件与数据库运行记录一致。
- 生成资源不是固定模板假数据。
- 失败、修订、通过三类决策都有可复现样例。

### M3：审核与反幻觉补强

目标：支撑比赛评分中的错误率和技术创新点。

任务：

- 接入 `primary_review_model` 和 `secondary_review_model`。
- 对事实准确、来源追溯、难度匹配、知识覆盖分别打分。
- 分差超过 10 分或一方通过一方失败时触发仲裁。
- 仲裁后仍不一致时标记 `manual_review_required`，不展示给学习者。

验收：

- `review_reports` 存储双模型结果、仲裁结果和最终结论。
- 未通过或人工复核资源不会出现在学习资源页的默认可学列表。
- 评测脚本可统计幻觉率相关指标。

### M4：反馈触发辅导闭环

目标：让“反馈 -> 更新”不是按钮提示，而是真正改变后续学习路径。

任务：

- `too_hard` 生成补救解释。
- `too_easy` 生成挑战任务。
- `confusing` 生成换一种讲法。
- `incorrect` 触发资源修订或复核。
- 报告页打开时，如果学习路径 `needs_refresh=true`，自动刷新或提示刷新。

验收：

- 资源页反馈后能在报告页看到路径或画像变化。
- Tutoring Agent 有独立输入、输出和运行记录。

### M5：评测脚本与指标面板

目标：形成可复现的比赛证明材料。

任务：

- 准备 50 个 evaluation cases。
- 覆盖 3 类资源、3 类画像、不同难度和不同审核结论。
- `test_script` 输出 hallucination_rate、difficulty_match、knowledge_coverage、path_order_accuracy（如实现）。
- 前端指标页读取或展示最新评测结果。

验收：

- 离线脚本一键运行。
- 指标达到：幻觉率 `<5%`，难度匹配 `>=85%`，核心知识覆盖 `>=90%`。

### M6：演示稳定与提交材料

目标：稳定、可讲、可复现。

任务：

- 编写演示脚本和 10 分钟操作路线。
- 准备一键初始化、重置演示数据、重建索引命令。
- 修复控制台阻塞错误、空状态和失败状态。
- 更新 README、API 文档、部署文档。

验收：

- 新环境按文档可启动。
- 演示路线不依赖临时手工修复。
- 提交包包含源码、设计实现方案、PPT、演示视频和评测结果。

## 4. 优先级调整

立即优先：

1. 统一前端访问入口，Docker 环境优先使用 `localhost:5173`。
2. 补齐知识库导入后的真实索引重建。
3. 把 Agent 运行记录与 SSE 事件打通。
4. 把资源审核从规则占位升级到双模型交叉验证。
5. 补齐 `test_script` 的 50 个评测样例。

暂缓：

- 完整账号体系和 JWT。
- 企业权限审批。
- `/api/v2`。
- Redis、Neo4j。
- 完整在线评测管理后台。
- 装饰性首页和营销页。

## 5. 演示路线

比赛演示优先采用以下路径：

```text
演示工作台
 -> 诊断测评：创建测评、填入演示答案、提交画像
 -> Agent 协作：启动协同流、观察节点状态
 -> 学习资源：查看三类资源、来源、审核状态
 -> 学习资源：提交 too_hard / too_easy / incorrect 反馈
 -> 学习报告：查看能力雷达、学习路径刷新
 -> 知识库管理：导入一个新知识点，展示待重建索引
 -> 指标页：展示评测目标和 test_script 输出
```

任何新功能都应优先服务这条路线。
