# 当前迭代计划

> 更新日期：2026-07-16
> 适用范围：比赛 MVP 开发期，主验证领域 `ai_app_dev`（人工智能应用开发实训）

## 1. 为什么调整原计划

原设计文档中的 9 周计划以“项目初始化 -> 数据模型与知识库 -> 诊断 -> Agent -> 可视化反馈 -> 报告评测”为主线，适合作为完整工程推进顺序。

但当前项目已经提前完成了部分前端演示闭环：

- Vue 前端应用壳、演示工作台、诊断测评、Agent 协作、学习资源、学习报告、指标页已形成可点击流程。
- 知识库管理页已支持单条手动导入知识点。
- 后端已有 `/api/v1` 路由骨架、统一响应、诊断、生成任务、资源、报告、知识库列表/检索/新增接口。
- 生成链路当前仍以 `demo_flow_service` 和规则化 MVP 图为主，LLM 双模型审核、真实异步任务和完整评测仍需补强。

因此迭代策略应调整为：

```text
前端演示闭环稳定 -> 后端真实能力补齐 -> 评测脚本量化 -> 演示材料打磨
```

目标不是重做计划，而是让后续开发围绕已经可见的产品流程继续填实。

## 3. 调整后的里程碑

### M0：前端演示闭环稳定（当前优先）

目标：保证评委和团队成员能从 `localhost:5173` 走完完整演示路线。

验收：

- 首页能清晰引导：诊断 -> Agent 协作 -> 资源 -> 反馈 -> 报告 -> 指标。
- 核心页面中文正常、无旧乱码、无营销落地页。
- 知识库页能新增知识点，并提示“待重建索引”。
- `npm run build` 通过。
- Docker 前端容器与本地访问地址一致，优先使用 `http://localhost:5173/`。

### M1：知识库与诊断数据补齐

目标：满足 MVP 数据规模和可解释性。

任务：

- 确认 `ai_app_dev` 至少 50 条真实知识点。
- 确认至少 60 道诊断题，选择题与简答题覆盖不同难度。
- 为知识点补齐分类、难度、标签、来源、内容。
- 实现知识变更后的索引重建流程：新增/修改知识点 -> 标记 `needs_reembedding` -> 重建 Chroma -> 清除标记。
- 知识关系表至少覆盖 prerequisite、dependent、related 三类关系。

验收：

- 领域配置页校验通过。
- 知识库检索能返回新增知识点。
- 手动导入后学习路径 `needs_refresh=true`。

### M2：后端闭环真实化

目标：把前端已经展示的流程逐步接成真实后端逻辑。

任务：

- 将 `demo_flow_service` 中的关键逻辑拆入正式 service。
- 生成任务支持真实异步状态流，SSE 事件来自任务状态和 `agent_runs`。
- Profile Analysis Agent、Knowledge Retrieval Agent、Content Generation Agent、Review and Validation Agent、Orchestrator Agent 输出结构化结果。
- 每个 Agent 写入 `agent_runs` 和 `agent_messages`。
- 资源生成必须包含来源引用。

验收：

- Agent 协作页展示的事件与数据库运行记录一致。
- 生成资源不是固定模板假数据。
- 失败、修订、通过三类决策都有可复现样例。

### M3：审核与反幻觉补强

目标：支撑比赛评分中的错误率和技术创新点。

任务：

- 接入 `primary_review_model` 和 `secondary_review_model`。
- 对事实准确、来源追溯、难度匹配、知识覆盖分别打分。
- 分差超过 10 分或一方通过一方失败时触发仲裁。
- 仲裁后仍不一致时标记 `manual_review_required`，不展示给学习者。

验收：

- `review_reports` 存储双模型结果、仲裁结果和最终结论。
- 未通过或人工复核资源不会出现在学习资源页的默认可学列表。
- 评测脚本可统计幻觉率相关指标。

### M4：统一反馈、证据判断与人工复核闭环

目标：让“反馈 -> 判断 -> 更新或不更新”通过同一 LangGraph 任务形成可追踪闭环，避免单次主观反馈直接改写画像。

任务：

- 自然语言反馈通过导学会话进入，快捷标签、评分和选中文本仅作为辅助证据。
- 第一次错误优先追问或提示，第二次仍错再生成补救解释。
- `too_easy` 先确认掌握证据，再决定挑战任务和画像更新。
- `incorrect` 触发资源复核，不修改学习者能力画像。
- 证据不足时保存 `no_change`、理由和即时解释，保留原画像、路径和资源。
- 证据充分时创建画像新版本，只刷新受影响知识点、路径节点和资源。
- 双模型持续分歧进入 `waiting_human`，管理员沿原 `task_id/thread_id` 恢复任务。
- 临时材料上传与隔离移入下一阶段 P1，不阻塞本轮 P0。

验收：

- 可复现画像更新与画像不更新两个分支，并展示证据和决定理由。
- Tutoring Agent 有独立输入、输出、会话消息和运行记录。
- 人工复核支持通过、要求修订和驳回，未解决资源对学习者不可见。

### M5：评测脚本与指标面板

目标：形成可复现的比赛证明材料。

任务：

- 准备 50 个 JSON evaluation cases，并可派生导出 XLSX。
- 覆盖 3 类资源、3 类画像、不同难度和不同审核结论。
- `test_script` 输出分子、分母、比率、失败 case ID、知识库/脚本版本和评测时间。
- 前端指标页读取或展示最新评测结果。

验收：

- 离线脚本一键运行。
- 指标达到：幻觉率 `<5%`，难度匹配 `>=85%`，核心知识覆盖 `>=90%`。

### M6：演示稳定与提交材料

目标：稳定、可讲、可复现。

任务：

- 编写演示脚本和 10 分钟操作路线。
- 准备一键初始化、重置演示数据、重建索引命令。
- 修复控制台阻塞错误、空状态和失败状态。
- 更新 README、API 文档、部署文档。

验收：

- 新环境按文档可启动。
- 演示路线不依赖临时手工修复。
- 提交包包含源码、设计实现方案、PPT、演示视频和评测结果。

## 4. 优先级调整

P0 已完成：

1. 统一八节点 LangGraph、Agent 运行记录与联合 SSE 事件。
2. 双模型独立审核、重新检索复审与人工 checkpoint 恢复。
3. 证据驱动的画像更新/不更新、资源版本链和局部影响范围。
4. 资源版本/导出、导学会话和管理员人工复核页面。
5. 50 个 JSON 评测案例及 JSON/Markdown/XLSX 报告。

P0.5 代码已完成：

1. 真实模型通道配置校验、Pydantic 输出校验和安全健康状态。
2. 知识点修改、增量 Chroma 重建、14 条 related 关系和局部影响标记。
3. baseline/live 分离评测、6/15/50 阶段控制及任务/Agent P50/P95。
4. Docker 一键启动/重置/验证/停止脚本和七分支验收程序。

待真实环境验收：

1. 用户填写接口地址、密钥和三个不同用途的模型名。
2. 依次执行 6、15、50 个真实模型案例并保存 live 报告。
3. 执行七分支验收；异常分支无法稳定复现时只允许使用带 `provider_mode=live` 的历史真实快照。
4. 完成端到端录像后进入 P1 临时材料上传与隔离。

暂缓：

- 完整账号体系和 JWT。
- 企业权限审批。
- `/api/v2`。
- Redis、Neo4j。
- 完整在线评测管理后台。
- 装饰性首页和营销页。

## 5. 演示路线

比赛演示优先采用以下路径：

```text
演示工作台
 -> 诊断测评：创建测评、填入演示答案、提交画像
 -> Agent 协作：启动协同流、观察节点状态
 -> 学习资源：查看三类资源、来源、审核状态
 -> 学习资源：提交 too_hard / too_easy / incorrect 反馈
 -> 学习报告：查看能力雷达、学习路径刷新
 -> 知识库管理：导入一个新知识点，展示待重建索引
 -> 指标页：展示评测目标和 test_script 输出
```

任何新功能都应优先服务这条路线。
