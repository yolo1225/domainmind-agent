## 1. 项目概述

### 1.1 项目背景

当前我国正处于新一轮科技革命和产业变革的历史交汇期。国家层面大力推进教育数字化战略，推动从"知识本位"向"能力本位"转型。大模型技术的爆发为实现教育数据的智能处理提供了可能。

然而，在人工智能、智能制造等前沿产业的人才培养场景中，存在三大核心矛盾：

| 矛盾 | 描述 |
|------|------|
| **"个性化"与"专业性"难以兼得** | 放宽约束迎合学习者 → 专业知识失去准确性；严格保证专业性 → 生成内容千人一面 |
| **大模型"幻觉"在严肃场景不可接受** | 通用大模型生成专业实操指导时，容易编造"九真一假"的错误内容 |
| **"书本知识"到"企业实操"的转化鸿沟** | 前沿产业技术迭代极快，传统静态教材无法跟上企业实战需求 |

### 1.2 项目目标

采用**多智能体协同决策**技术路线，构建一套面向垂直领域的知识个性化生成系统，实现：

1. **精准画像**：自动诊断学习者的知识盲区与能力层级
2. **个性化路径与资源匹配**：结合学习者先验知识或能力画像与垂直领域知识库，生成学习顺序、节点资源和可解释的推荐理由
3. **多Agent协同生成**：至少3个独立Agent分工协作，生成高可靠性的个性化学习资源
4. **破解"幻觉"**：通过交叉验证机制，将知识错误率控制在5%以下
5. **可视化决策**：为培训者/学习者提供直观的学习分析报告与路径规划
6. **反馈驱动更新**：根据答题和资源反馈判断画像是否发生有效变化，并有条件地触发补救解释、进阶挑战、资源复核及学习路径更新

### 1.3 项目范围

| 维度 | 范围说明 |
|------|---------|
| **主验证领域** | 人工智能应用开发实训，覆盖大模型应用开发、RAG知识库构建、Prompt工程、Agent编排等典型技能任务 |
| **领域迁移能力** | 系统架构支持通过替换知识库、知识点体系、能力标准、审核规则和资源模板迁移到其他垂直领域 |
| **知识库规模** | 至少1个人工智能应用开发专业知识库，≥50条真实知识点（项目增强要求） |
| **诊断题库规模** | ≥60道真实诊断题，覆盖选择题、简答题和不同难度（项目增强要求） |
| **目标用户** | 企业一线技术人员、高校学生、职业培训学员 |
| **输出资源** | 3种形态——定制化讲义、实操指南、分阶测试题 |
| **核心指标** | 知识错误率<5%，难度匹配准确率≥85%，知识点覆盖率≥90% |

> **需求来源说明**：官方硬性或满分要求包括至少1个垂直领域知识库切片、至少3个职责明确的Agent、至少3种资源、完整反馈闭环、不少于2组差异化学情数据（满分测试方案为≥3组）及可复现质量指标。≥50条知识点、≥60道诊断题、双模型仲裁和SSE属于本项目为提高演示与评测可信度设置的增强要求或技术实现，不表述为官方指定技术方案。

## 2. 用户角色分析

### 2.1 角色定义

| 角色 | 描述 | 核心诉求 |
|------|------|---------|
| **学习者（Learner）** | 需要获取个性化学习资源的终端用户 | 获得匹配自身水平的精准学习内容，了解自己的知识盲区 |

## 3. 功能性需求

### 3.1 学情画像构建模块（F1）

**对应赛题要求(1)**：聚焦垂直领域的典型技术培训场景，精准区分不同画像学习者的能力差异。

#### F1.1 学习者基础信息采集

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F1.1.1 | 系统应支持录入学习者的基本信息：姓名、学习背景（学历/专业）、目标领域、相关经验年限 | P0 |
| F1.1.2 | 系统应支持学习者自主选择学习方向标签（如："Python机器学习"、"工业机器人运维"等） | P0 |
| F1.1.3 | 系统应支持培训者批量导入学习者信息（Excel/CSV格式） | P1 |

#### F1.2 初始能力诊断测试

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F1.2.1 | 系统应根据学习者选择的方向，从知识库中自动抽取≥10道诊断题目组成初始测试卷 | P0 |
| F1.2.2 | 题目类型应包含选择题和简答题，覆盖基础理论、实操场景两个维度 | P0 |
| F1.2.3 | 学习者完成测试后，系统应自动批改选择题，简答题由AI辅助评分并给出评语 | P0 |
| F1.2.4 | 系统应记录学习者的答题路径（每道题的作答时间、修改次数、是否跳过） | P1 |
| F1.2.5 | `ai_app_dev`诊断题库应包含≥60道真实题目，覆盖选择题、简答题及不同难度；单次诊断仍按F1.2.1抽取≥10题 | P0 |

#### F1.3 知识盲区识别

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F1.3.1 | 系统应基于答题记录，自动将错题关联到知识图谱中的具体知识点节点 | P0 |
| F1.3.2 | 系统应输出结构化的"知识盲区清单"，包含：薄弱知识点名称、薄弱程度（1-5级）、关联的前置知识点 | P0 |
| F1.3.3 | 系统应支持区分"完全未掌握"和"部分掌握但易混淆"两种盲区类型 | P1 |

#### F1.4 学情画像可视化

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F1.4.1 | 系统应生成学习者的能力雷达图，维度≥5个（如：理论基础、实操能力、问题解决、知识广度、学习速度） | P0 |
| F1.4.2 | 系统应展示"知识掌握热力图"——在知识图谱上标注已掌握/薄弱/未接触的知识区域 | P1 |
| F1.4.3 | 系统应支持画像的导出（PDF格式） | P1 |

### 3.2 Multi-Agent协同生成模块（F2）⭐ 核心

**对应赛题要求(2)**：至少3个以上具有明确职责分工的Agent，实现"画像→检索→生成→校验→决策"的协同闭环，动态生成至少3种形态的个性化学习资源。

#### F2.1 Agent角色定义（官方至少3个，项目MVP采用6个核心角色）

| 编号 | Agent名称 | 核心职责 | 输入 | 输出 | 优先级 |
|------|-----------|---------|------|------|--------|
| F2.1.1 | **学情分析Agent** | 精准诊断学习者的知识强项、弱项与技术盲区，生成结构化学情报告 | 学习者基本信息 + 答题记录 | 知识盲区清单 + 能力等级评估 + 学习风格标注 | P0 |
| F2.1.2 | **知识检索Agent** | 从知识库中多路召回与学习者当前需求最匹配的知识点 | 学情分析结果 + 学习目标 | 知识点列表（含匹配度排序） + 关联资源索引 | P0 |
| F2.1.3 | **内容生成Agent** | 基于画像与检索结果，生成3种形态的个性化学习资源 | 学习者画像 + 知识点列表 + 资源类型要求 | 定制化讲义 + 实操指南 + 分阶测试题 | P0 |
| F2.1.4 | **审核校验Agent** | 对生成内容进行多维交叉验证，过滤"幻觉"和错误 | 生成Agent的输出 + 原始知识库 | 校验报告（通过/需修改/不通过） + 修改建议 | P0 |
| F2.1.5 | **协调编排Agent** | 统筹调度其他Agent的协作流程，执行下一步决策，处理异常与冲突仲裁 | 学习者请求 + 上下文状态 | 任务调度方案 + 决策结果 + 流程状态 | P0 |
| F2.1.6 | **交互导学Agent** | 通过自然语言对话识别学习者反馈意图，进行动态追问、启发式提示，并判断是否需要更新画像或资源 | 学习者当前回答 + 对话反馈 + 学情画像 + 资源学习记录 + 临时材料上下文 | 结构化反馈意图 + `profile_update_required` + 画像变更证据 + 推荐动作 + 受影响路径节点/资源 | P0 |

#### F2.2 Agent协同编排

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F2.2.1 | 系统应实现"画像→检索→生成→校验→决策"的**完整闭环流程**，支持自动流转与人工干预两种模式 | P0 |
| F2.2.2 | 每个Agent应具备独立的系统提示词（System Prompt）、独立的工具集和独立的工作记忆（上下文窗口） | P0 |
| F2.2.3 | Agent之间的通信应基于标准化的消息格式，包含：发送方ID、接收方ID、消息类型、载荷内容、时间戳、会话ID | P0 |
| F2.2.4 | 协调Agent应支持异常处理：当某个子Agent返回异常结果时，触发重试或降级策略 | P1 |
| F2.2.5 | 系统应支持Agent协同过程的**实时可视化**——前端以流程图形式展示每个Agent的工作状态（等待中/执行中/已完成/异常） | P0 |
| F2.2.6 | 系统应记录完整的Agent协同日志，用于后续分析与调试 | P1 |
| F2.2.7 | 系统应支持启发式交互导学：当学习者回答错误或表达困惑时，导学Agent应先追问或提示，再决定是否生成降维解释、补充案例或进阶任务 | P0 |

#### F2.3 个性化资源生成（3种形态）

| 编号 | 资源形态 | 需求描述 | 优先级 |
|------|---------|---------|--------|
| F2.3.1 | **定制化讲义** | 根据学习者水平调整讲解深度，根据学习风格调整案例类型（理论型/实操型），包含：知识点讲解 + 匹配案例 + 关键要点总结 | P0 |
| F2.3.2 | **实操指南** | 提供分步骤操作指导，包含：环境准备、操作步骤（每步含代码/命令/截图）、常见问题排查、预期结果验证 | P0 |
| F2.3.3 | **分阶测试题** | 生成3个难度层级的测试题——基础巩固（覆盖盲区）、能力提升（关联知识点）、挑战突破（综合应用），每题含解析 | P0 |
| F2.3.4 | 所有生成资源必须标注**信息来源**（引用了知识库中哪些条目），以便追溯验证 | P0 |
| F2.3.5 | 所有生成资源必须标注**难度等级**和**适配的学习者画像类型** | P1 |
| F2.3.6 | **资源导出** | 定制化讲义和实操指南应支持Markdown或PDF导出，导出内容保留难度、知识来源、审核状态和生成时间 | P0 |
| F2.3.7 | **测试题导出** | 分阶测试题应支持无答案学习者版和含答案解析版导出 | P0 |
| F2.3.8 | **学习包导出** | 系统宜支持导出包含学习路径、节点资源、来源和审核摘要的完整学习包 | P1 |

#### F2.4 动态更新机制

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F2.4.1 | 系统应同时采集快捷反馈标签和自然语言对话反馈，快捷标签至少包括"太简单"、"太难"、"有帮助"和"有错误" | P0 |
| F2.4.2 | 导学Agent应将反馈识别为结构化意图，并决策补救解释、换一种讲法、进阶挑战、错误复核或暂不调整 | P0 |
| F2.4.3 | 系统应结合答题结果、对话反馈和学习行为，判断`profile_update_required`；单次主观反馈不得在缺少证据时直接覆盖能力画像 | P0 |
| F2.4.4 | 当画像发生有效变化时，系统应保存画像前后版本、变化维度、证据、置信度、触发反馈和决策理由 | P0 |
| F2.4.5 | 画像变化后，系统只更新受影响的学习路径节点和资源，并保留未受影响内容，避免无意义的全量重新生成 | P0 |
| F2.4.6 | 当画像未发生有效变化时，系统应保留原画像、学习路径和资源，仅返回追问、解释或提示，并记录不更新理由 | P0 |
| F2.4.7 | 学习者完成一轮学习后再次测试时，系统应按相同证据规则更新画像并重新评估学习路径 | P0 |

#### F2.5 审核、交叉验证与仲裁

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F2.5.1 | 审核Agent应分别检查事实准确性、来源可追溯性、难度匹配度和核心知识覆盖率，并输出结构化分项评分与证据 | P0 |
| F2.5.2 | `primary_review_model`与`secondary_review_model`都必须检查事实和来源，审核结果应独立保存，不得由单次模型调用伪造双通道结果 | P0 |
| F2.5.3 | 两个审核模型总分相差超过10分，或一方通过而另一方不通过时，系统应重新检索来源并再次审核 | P0 |
| F2.5.4 | 再次审核后仍无法达成一致时，资源应标记为`manual_review_required`，不得进入学习者默认可见资源列表 | P0 |
| F2.5.5 | 审核报告应保存两路评分、引用证据、分歧原因、仲裁过程、最终结论和审核规则版本 | P0 |
| F2.5.6 | Agent协同页应可视化展示至少一个自我博弈案例，包括两路审核结论、分歧点、重新检索证据、仲裁前后评分和最终决策 | P0 |

### 3.3 智能决策与反馈模块（F3）

**对应赛题要求(3)**：支持生成可可视化的个性化学习资源匹配报告，支持动态更新机制，支持Multi-Agent协同的"纠错""更新""挑战任务"。

#### F3.1 学习分析报告

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F3.1.1 | 系统应生成**知识盲区定位图**——在知识图谱上高亮标注学习者的薄弱区域，用颜色深浅表示薄弱程度 | P0 |
| F3.1.2 | 系统应生成**资源难度匹配雷达图**——展示学习者当前能力等级与已生成资源的难度匹配关系 | P0 |
| F3.1.3 | 系统应生成**学习路径规划图**——以时间轴或流程图展示推荐顺序与里程碑；每个节点应包含目标知识点、对应资源、画像弱项依据、推荐理由、完成条件和下一步决策 | P0 |
| F3.1.4 | 培训者应可查看**群体学情概览**——全班/全组学习者的能力分布统计、共性薄弱点分析 | P1 |

#### F3.2 反馈机制

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F3.2.1 | 学习者可通过快捷标签、星级评价和自然语言对话反馈资源难度、帮助程度、困惑点或错误 | P0 |
| F3.2.2 | 学习者可标记生成内容中的具体错误（选中文本 + 描述问题），提交后触发审核Agent重新校验 | P1 |
| F3.2.3 | 系统应支持3种Multi-Agent协同场景：**知识纠错**（发现错误→检索Agent复核→生成Agent修正→校验Agent验证）、**知识更新**（新知识入库→关联Agent更新图谱→通知受影响的学习路径）、**挑战性任务**（学习者主动请求更高难度→重新评估→生成进阶资源） | P0 |

## 4. 非功能性需求

### 4.1 性能指标

| 编号 | 指标项 | 目标值 | 测量方法 |
|------|--------|--------|---------|
| NF1.1 | 完整生成流程响应时间 | ≤ 30秒 | 从提交学习者画像到3种资源全部生成完毕的端到端耗时 |
| NF1.2 | 单Agent调用响应时间 | ≤ 10秒 | 单个LLM API调用的平均耗时 |
| NF1.3 | 系统并发支持 | ≥ 5个并发用户 | 同时提交请求不出现服务降级 |

### 4.2 质量指标（来自赛题评分标准）

| 编号 | 指标项 | 目标值 | 验收说明 |
|------|--------|--------|---------|
| NF2.1 | **知识错误率（幻觉率）** | **< 5%** | 以可核验事实声明为统计单位，同时报告知识保真率`1-幻觉率` |
| NF2.2 | **画像-资源难度匹配准确率** | **≥ 85%** | 与评测用例预设的人工金标准难度标签比较 |
| NF2.3 | **核心知识点覆盖率** | **≥ 90%** | 以当前生成任务应覆盖的目标核心知识点集合为分母 |
| NF2.4 | 评测样例数量 | ≥ 50组 | 测试用例不足50组将影响官方实用价值评分 |

#### 4.2.1 指标计算口径

| 指标 | 计算公式 | 统计约束 |
|------|---------|---------|
| 幻觉率 | 错误或无法由引用来源支持的事实声明数 / 可核验事实声明总数 | 事实、操作步骤、代码行为和预期结果均可作为声明；无法判定的声明单独报告，不得直接按正确处理 |
| 难度匹配准确率 | 与人工金标准难度标签一致的资源数 / 参与评测的资源总数 | 每个evaluation case预先标注画像等级、目标难度和判定规则；人工分歧应仲裁并留痕 |
| 核心知识点覆盖率 | 已被资源正确覆盖的目标核心知识点数 / 当前任务应覆盖的核心知识点总数 | 分母来自诊断结果、学习目标和检索计划，不使用整个领域知识体系作为单份资源的分母 |

三项指标应基于版本固定的evaluation cases和知识库运行。`test_script`必须输出分子、分母、最终比率、失败样例ID、知识库版本和评测时间，前端指标页不得用展示值替代离线评测结果。

### 4.3 安全性

| 编号 | 需求描述 |
|------|---------|
| NF3.1 | 用户个人身份数据应进行脱敏处理（姓名可用代号替代，联系方式不采集） |
| NF3.2 | 学习者答题记录、学情画像等敏感数据不得在日志中明文输出 |
| NF3.3 | 知识库内容应进行来源标注，避免使用未授权的版权内容 |

### 4.4 可靠性

| 编号 | 需求描述 |
|------|---------|
| NF4.1 | LLM API调用失败时，系统应自动重试（最多3次），重试间隔递增（1s/3s/5s） |
| NF4.2 | 单次Agent调用超时（>30秒）应触发降级策略——使用缓存结果或通知用户稍后重试 |
| NF4.3 | 知识库数据应定期备份（至少每日一次） |

### 4.5 可维护性

| 编号 | 需求描述 |
|------|---------|
| NF5.1 | 各Agent应解耦设计，新增Agent或替换Agent实现不应影响其他Agent |
| NF5.2 | LLM提供商切换（如从GPT-4切换到智谱GLM-4）应通过配置完成，无需修改业务代码 |
| NF5.3 | 代码应包含关键模块的单元测试（Agent调度逻辑、知识检索逻辑、校验逻辑） |

## 5. 数据需求

知识条目（KnowledgeItem）

│   └── 测试题集合
├── created_at: 创建时间
└── updated_at: 更新时间
```

```
学习者（Learner）

├── target_domain: 目标学习领域

├── profile_change: 最近画像变更摘要
│   ├── changed_dimensions: 变化维度
│   ├── evidence_refs: 答题、反馈或行为证据ID
│   ├── confidence: 变更置信度
│   ├── trigger_feedback_id: 触发反馈ID
│   └── decision_reason: 更新或不更新理由

## 7. 约束与假设

### 7.1 技术约束

| 编号 | 约束项 | 说明 |
|------|--------|------|
| C1 | **避免伪Multi-Agent** | 系统不应仅表现为同一模型的无状态连续问答；每个Agent应具备明确职责、独立输入输出、可追踪状态、专属工具或校验规则 |
| C2 | **必须使用真实领域数据** | 知识库不能是虚构内容，知识点必须来自真实学科/行业标准 |
| C3 | **不得侵犯第三方知识产权** | 知识库内容应自建或使用开源授权内容 |
| C4 | **完整闭环不可缺失** | 系统必须覆盖画像、诊断、检索、生成、审核、决策、反馈和更新全流程，不可只做其中的片段 |
| C5 | **画像更新必须有证据** | 系统只能依据可追踪的答题、对话或学习行为证据更新画像，并记录更新或不更新理由 |
| C6 | **上传材料必须隔离** | 学习者上传材料默认仅用于当前任务上下文，不得自动进入正式知识库或直接改写学习者画像 |

### 7.2 假设条件

| 编号 | 假设项 |
|------|--------|
| A1 | 假设团队具备基本的Web全栈开发能力（前后端 + 数据库） |
| A2 | 假设LLM API（GPT-4或同等模型）可稳定调用，预算充足 |
| A3 | 假设可在规定时间内获取至少1个垂直领域的≥50条真实专业知识 |
| A4 | 假设团队成员对至少1个垂直领域（如AI/编程）有足够专业知识来构建和审核知识库 |
